# CI/CD 2026 - ODOS (Contabo-ready)

Ce workflow est optimise pour votre monorepo ODOS avec:

- backend Symfony (`odos-back`)
- frontend Expo/React Native (`odos-front`)
- deploiement prod sur VPS Contabo via Docker Compose

## 1) Pipelines inclus

- `.github/workflows/ci.yml`
  - detection de changements (`backend`, `frontend`, `infra`)
  - tests backend (`composer test`)
  - analyse statique backend (`composer analyse:ci`)
  - tests frontend (`pnpm run lint`, `pnpm run test:ci`)
  - coverage backend + frontend sur `push main` (artifacts HTML)
  - build image Docker backend cible `prod`
- `.github/workflows/deploy-prod.yml`
  - deploy auto apres CI reussie sur `main`
  - deploy manuel possible (`workflow_dispatch`)
  - migration Doctrine en fin de deploiement
- `docker-compose.prod.yml`
  - force `php` en target `prod`
  - force `APP_ENV=prod`
  - desactive LLM local par defaut (`LLM_ENABLED=false`)
  - isole le service `llm` via profile (`llm`)

## 2) Secrets GitHub a creer

Dans `Settings > Secrets and variables > Actions`:

- `PROD_HOST`: IP ou DNS du VPS
- `PROD_USER`: utilisateur SSH
- `PROD_SSH_KEY`: cle privee SSH (format OpenSSH)
- `PROD_PORT`: port SSH (souvent `22`)
- `PROD_APP_PATH`: chemin absolu du repo sur le serveur (ex: `/opt/odos`)

Secrets optionnels (notifications):

- `SLACK_WEBHOOK_URL`: webhook Slack entrant
- `DISCORD_WEBHOOK_URL`: webhook Discord

Variables GitHub optionnelles (`Settings > Secrets and variables > Actions > Variables`):

- `ENABLE_PROD_DEPLOY` (defaut recommande: `false` tant que le serveur n'est pas pret)
- `PROD_HEALTHCHECK_URL` (URL sante, ex: `https://api.ton-domaine.com/health`)
- `PROD_HEALTHCHECK_RETRIES` (defaut: `12`)
- `PROD_HEALTHCHECK_DELAY_SECONDS` (defaut: `5`)

## 2.1) Versions d'actions utilisees (etat actuel)

Dans `.github/workflows/ci.yml`:

- `actions/checkout@v6`
- `actions/setup-node@v6`

## 3) Pre-requis serveur Contabo

- Docker + Docker Compose plugin installes
- repo clone dans `PROD_APP_PATH`
- variables d'environnement prod renseignees (fichier `.env` serveur ou mecanisme equivalent):
  - `APP_SECRET`
  - `DATABASE_URL`
  - `REDIS_URL`
  - `APP_HTTP_PORT` (optionnel, `80` conseille en prod)
  - `CORS_ALLOW_ORIGIN` (si necessaire)
  - secrets JWT/OAuth/Twilio utilises par l'app

### Variables minimales requises par la CI backend

Le backend CI execute `composer install` puis `cache:clear` (auto-scripts Symfony).  
Pour eviter les echecs `Environment variable not found`, la CI injecte notamment:

- `APP_ENV=test`
- `DATABASE_URL=postgresql://postgres@127.0.0.1:5432/odos_test?...`
- `REDIS_URL=redis://127.0.0.1:6379`
- `LLM_ENABLED=false`
- `DEFAULT_URI=http://localhost` (ecrit dans `.env` de job)

Note: le job `backend-static-analysis` doit avoir **le meme env runtime** que les jobs backend de test/coverage.

## 4) Strategy recommandee (best practices 2026)

- protection de branche `main`:
  - merge uniquement si `CI` verte
  - reviews obligatoires
- deploiement:
  - auto uniquement depuis `main`
  - environnement GitHub `production` avec approbation manuelle recommandee
- observabilite:
  - healthcheck applicatif
  - logs centralises (au minimum retention)
  - alertes uptime/erreur 5xx
- sauvegardes:
  - DB quotidienne + retention
  - test de restauration periodique

## 5) Commandes utiles en prod

- Deployer manuellement:
  - `docker compose -f docker-compose.yml -f docker-compose.prod.yml up -d --build --remove-orphans`
- Migrer la base:
  - `docker compose exec -T php php bin/console doctrine:migrations:migrate --no-interaction --env=prod`
- Purge rĂ©tention RGPD (cron quotidien recommandĂ©) :
  - `scripts/prod-cron-purge.sh` ou `app:data-retention:purge --env=prod`
  - Voir `docs/LOG_RETENTION.md`
- Variables JWT prod : `JWT_TOKEN_TTL=900`, `JWT_REFRESH_TTL=2592000`, `CORS_ALLOW_ORIGIN` sans wildcard
- Activer le profil LLM local (si vous hebergez Ollama):
  - `docker compose --profile llm -f docker-compose.yml -f docker-compose.prod.yml up -d`

## 6) Ce que fait la phase 2 (rollback + healthcheck + alerting)

- `deploy-prod.yml` deploie puis lance `doctrine:migrations:migrate`
- si `PROD_HEALTHCHECK_URL` est configuree:
  - le workflow teste l'URL plusieurs fois (`RETRIES` x `DELAY`)
  - si echec: rollback automatique sur le commit precedent puis redeploiement
  - le job echoue pour garder une trace claire
- en cas d'echec de deploy:
  - notification Slack si `SLACK_WEBHOOK_URL` existe
  - notification Discord si `DISCORD_WEBHOOK_URL` existe

## 7) Protection de l'environnement `production` (important)

Dans `Settings > Environments > production`:

1. Activez `Required reviewers` (au moins 1 approbation)
2. Optionnel: ajoutez `Wait timer` (ex: 5 min) pour laisser le temps d'annuler
3. Placez les secrets prod directement dans cet environnement si vous voulez les isoler du reste

## 8) Decoupage propre (etat actuel)

### Ce qui est finalise maintenant (sans serveur)

- CI complete active via `.github/workflows/ci.yml`
- tests/lint/build verifies a chaque PR et push `main`
- coverage backend/frontend generee sur `main`
- doc et compose prod prets

### Ce qui est en attente serveur

- `deploy-prod.yml` est protege par `ENABLE_PROD_DEPLOY`
- tant que `ENABLE_PROD_DEPLOY=false`, aucun deploy auto ne part
- une fois serveur pret:
  1. creer les secrets `PROD_*`
  2. configurer `PROD_HEALTHCHECK_URL`
  3. passer `ENABLE_PROD_DEPLOY=true`
  4. lancer un deploy manuel (`workflow_dispatch`) pour validation
