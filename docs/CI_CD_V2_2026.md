# CI/CD — ODOS

Comment la CI et le déploiement sont câblés pour le monorepo (Symfony + Expo) sur Contabo.

Index : [docs/README.md](README.md)

---

## Ce qui tourne

### `.github/workflows/ci.yml`

À chaque PR et push sur `main` :

- détection des changements (`backend`, `frontend`, `infra`)
- backend : PHPUnit + PHPStan
- frontend : lint + tests Jest
- sur push `main` uniquement : coverage back/front (artifacts HTML)
- build de l'image Docker backend (cible `prod`)

Un job `deploy` existe aussi dans ce workflow (secrets `VPS_HOST`, `VPS_USER`, `VPS_SSH_KEY`) — déploiement simple après CI verte. Si tu préfères le deploy avec rollback, utilise plutôt `deploy-prod.yml` (voir ci-dessous).

### `.github/workflows/deploy-prod.yml`

Deploy plus complet, déclenché :

- manuellement (`workflow_dispatch`)
- ou automatiquement après CI réussie sur `main`

**Condition :** la variable `ENABLE_PROD_DEPLOY` doit valoir `true`. Tant qu'elle est à `false`, rien ne part.

Ce workflow fait :

- backup DB avant deploy
- `git pull` + `docker compose` prod
- migrations Doctrine
- healthcheck (si `PROD_HEALTHCHECK_URL` est renseignée)
- rollback auto si le healthcheck échoue
- notification Slack/Discord en cas d'échec

### `docker-compose.prod.yml`

- PHP en target `prod`, `APP_ENV=prod`
- LLM local désactivé par défaut (`LLM_ENABLED=false`)
- service `llm` isolé dans un profile Docker

### EAS Workflows (mobile Android)

Projet Expo : `@manuel971/odos` (`odos-front`, `eas.json`).

| Fichier | Déclenchement |
|---------|----------------|
| `odos-front/.eas/workflows/build-android-preview.yml` | Push sur `main` si `odos-front/**` change |
| `odos-front/.eas/workflows/build-android-production.yml` | Manuel (`eas workflow:run` ou dashboard) |

Commandes locales (depuis `odos-front`) :

```bash
pnpm run eas:workflow:preview
pnpm run eas:workflow:prod
```

**Activation GitHub :** [expo.dev](https://expo.dev) → projet **odos** → **GitHub** → connecter `ManuelB971/ODos`. Vérifier que la racine de l’app pointe vers `odos-front` si le dashboard le propose (monorepo).

**Premier build :** un `eas build --platform android --profile preview` interactif reste recommandé une fois pour les credentials Android.

Profils et `EXPO_PUBLIC_API_URL` : `odos-front/eas.json`. Détail APK : [PROD_SANS_DOMAINE.md](PROD_SANS_DOMAINE.md).

## Secrets GitHub

`Settings → Secrets and variables → Actions`

| Secret | Contenu |
|--------|---------|
| `PROD_HOST` | IP ou DNS du VPS |
| `PROD_USER` | Utilisateur SSH |
| `PROD_SSH_KEY` | Clé privée OpenSSH |
| `PROD_PORT` | Port SSH (souvent `22`) |
| `PROD_APP_PATH` | Chemin du repo sur le serveur (ex. `/opt/odos`) |

Optionnels — notifications :

- `SLACK_WEBHOOK_URL`
- `DISCORD_WEBHOOK_URL`

Pour le deploy simple dans `ci.yml` : `VPS_HOST`, `VPS_USER`, `VPS_SSH_KEY`.

### Variables (pas secrets)

| Variable | Défaut | Rôle |
|----------|--------|------|
| `ENABLE_PROD_DEPLOY` | `false` | Bloque le deploy auto tant que le serveur n'est pas prêt |
| `PROD_HEALTHCHECK_URL` | `https://api.odos-api.com/api/categories` | URL qui doit répondre **2xx–3xx** (pas `/api` seul → 401) |
| `PROD_HEALTHCHECK_RETRIES` | `12` | Nombre de tentatives |
| `PROD_HEALTHCHECK_DELAY_SECONDS` | `5` | Pause entre deux tentatives |

## Prérequis serveur

- Docker + Compose plugin
- repo cloné dans `PROD_APP_PATH`
- `.env` prod rempli (`APP_SECRET`, `DATABASE_URL`, `REDIS_URL`, JWT, CORS, Twilio si MFA SMS…)

Modèle : `odos-config/env.prod.example`

### Env injectée en CI backend

Pour que `composer install` et `cache:clear` passent sans erreur :

```
APP_ENV=test
DATABASE_URL=postgresql://postgres@127.0.0.1:5432/odos_test?...
REDIS_URL=redis://127.0.0.1:6379
LLM_ENABLED=false
DEFAULT_URI=http://localhost
```

Le job PHPStan doit avoir le même env que les jobs de test.

## Commandes prod utiles

Deploy manuel :

```bash
docker compose -f docker-compose.yml -f docker-compose.prod.yml up -d --build --remove-orphans
```

Migrations :

```bash
docker compose exec -T php php bin/console doctrine:migrations:migrate --no-interaction --env=prod
```

Purge rétention RGPD (cron quotidien recommandé) :

```bash
scripts/prod-cron-purge.sh
# ou : app:data-retention:purge --env=prod
```

Voir [LOG_RETENTION.md](LOG_RETENTION.md).

JWT prod : `JWT_TOKEN_TTL=900`, `JWT_REFRESH_TTL=2592000`, `CORS_ALLOW_ORIGIN` sans wildcard.

LLM local en prod (si tu héberges Ollama) :

```bash
docker compose --profile llm -f docker-compose.yml -f docker-compose.prod.yml up -d
```

## Mettre le deploy en route

1. Créer les secrets `PROD_*` (ou `VPS_*` si tu restes sur le job dans `ci.yml`)
2. Configurer `PROD_HEALTHCHECK_URL` → `https://api.odos-api.com/api/categories` (ou `/api/health` après merge du `HealthController`). **Ne pas** utiliser `/api` seul (401 = échec du deploy).
3. Dans `Settings → Environments → production` : activer `Required reviewers` si tu veux une validation humaine
4. Passer `ENABLE_PROD_DEPLOY=true`
5. Lancer un `workflow_dispatch` :
   - **`target: all`** — API + web app
   - **`target: web`** — uniquement `app.odos.world` (sans redéployer l'API)
   - **`target: api`** — uniquement le backend

Tant que `ENABLE_PROD_DEPLOY=false`, la CI tourne mais ne déploie pas via `deploy-prod.yml`.
